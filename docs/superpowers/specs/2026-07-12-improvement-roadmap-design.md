# 개선 로드맵 설계 — 팩트 중심 내부 도구로의 재정립

작성일: 2026-07-12 · 승인: 사용자 확정 (브레인스토밍 세션)

## 제품 정체성 (확정 사항)

| 항목 | 결정 |
|---|---|
| 최종 목적지 | **회사/팀 내부 도구** (외부 상용 서비스 아님) |
| 사용 규모 | 2~5명, 주 몇 회 — 현 서버(Lightsail 1GB) 유지, 증설·관리자 기능 불필요 |
| 보고서 용도 | 내부 검토 + 외부 제출 겸용 → **내보낸 PPT는 PowerPoint에서 개별 요소 편집 가능해야 함** (기존 원칙 유지) |
| 디자인 기준 | 팀의 기존 수작업 보고서 제공 예정 — **그 문서가 디자인 정답지** |
| 점수 철학 | **종합점수 축소, 팩트(거리·개수·시간) 중심** 재편 |
| 진행 순서 | A안: 신뢰성 → 보고서 → 사용성 (사용자 선택) |
| 모델 분담 | 코드 작업 = Sonnet 에이전트 / 계획·검토·검증 = Fable |

4대 불만(사용자 확인): 데이터 신뢰성, PPT 디자인 품질, 분석 점수 근거, 속도·사용성 — 전부 해당.

## 1단계 — 데이터 신뢰성: "같은 곳을 분석하면 언제나 같은 결과"

**문제 근거(2026-07-12 실측):** 동일 좌표 연속 분석에서 POI 210→62→0개. 원인: ① 외부 수집 결과가 서버 메모리 10분 캐시에만 존재(재시작·HMR 시 소실) ② Overpass 레이트리밋 시 카테고리 단위 무통보 소실.

### 1-1. 수집 결과 영구 캐시 (SQLite)
- 성공 수집분을 기존 better-sqlite3 DB(.cache/site-analysis.db)에 저장, 기본 보관 7일
- 캐시 키: 좌표(반올림)+반경+카테고리. 적중 시 수초 내 응답
- UI에 "이 데이터는 YYYY-MM-DD 수집본" 표기 + [새로 수집] 버튼(강제 갱신)
- 서버 메모리 TTL 캐시는 1차 캐시로 유지(계층화)

### 1-2. 부분 실패 투명화
- 8개 카테고리(지하철·학교·공원·산·아파트·오피스텔·기타주거·정비사업)별 수집 성공/실패/출처/수집시각을 분석 결과에 포함
- 실패 시: 사이드바 배지 "⚠️ {카테고리} 수집 실패 — [재시도]", 실패 카테고리 단독 재시도 지원
- PPT 표지 또는 출처 슬라이드에 누락 카테고리 명시("본 보고서는 학교 데이터가 누락된 상태로 생성됨")

### 1-3. 외부 소스 예절
- Overpass 요청 간 최소 간격(스로틀), 혼잡 시 미러 엔드포인트(lz4.overpass-api.de — CSP에 이미 허용됨) 폴백
- 서버 재시도 백오프 유지(기존 928ea9f·0c19427 로직 위에 구축)

### 1단계 완료 기준 (수용 테스트)
1. 동일 장소 연속 5회 분석 → POI 개수 5회 모두 동일
2. 외부 소스 차단 상태에서 분석 → 화면·보고서 양쪽에 누락 카테고리가 정확히 표기
3. 캐시 적중 재분석 소요 ≤ 5초

## 2단계 — 팩트 중심 보고서 + 회사 템플릿

**전제:** 사용자가 기존 수작업 보고서(PPT/PDF) 제공 → 레이아웃·서체·색·슬라이드 순서 분석 후 세부 설계 확정.

### 2-1. 팩트 시트 재편
- 카테고리 슬라이드의 주인공을 팩트로: 교통(최근접 역명·직선거리 m·도보시간·노선 수), 교육(초중고 개수·최근접 거리), 자연(최근접 공원/산·거리·개수), 주거(단지 수·세대수 합·분양 현황)
- **모든 수치에 출처+수집 기준일 표기** (1-2의 메타데이터 활용)
- 도보시간은 직선거리 기반 환산(80m/분) 명시 — 경로 탐색 도입은 범위 외

### 2-2. 종합점수 축소
- 점수 대시보드 슬라이드를 기본 구성에서 제외, "종합 의견" 슬라이드의 보조 지표로 이동
- 내보내기 옵션에서 점수 슬라이드 on/off 선택 가능

### 2-3. 템플릿 적용
- 제공받은 보고서 기준으로 SlideSpec(미리보기=내보내기 공유 모델) 디자인 교체
- 개별 요소 편집성 원칙 유지, qa/validate-preview-parity.mjs로 미리보기-PPT 일치 검증

### 2-4. 확정된 부속 결정 (2026-07-12 사용자 확정)
- **PPT 속 지도**: 집GPT식 표현을 **역사도식선·역명/노선 배지 수준까지 반영** (정제 데이터 osm-subway.json 기반, 웹과 통일). 출입구 마커·커넥터는 PPT 축척(반경 전체 뷰)에서 노이즈이므로 제외. 미리보기(canvas)·내보내기(pptx) 양쪽 렌더러에 동일 적용 — 2단계 템플릿 작업에 포함
- **PPT 프리셋 3종 복원 안 함**: 회사 템플릿 단일 디자인에 집중. 백업은 Obsidian(Architecture/server-only-ppt-design-config.ts.bak)에 보관, 필요 시 복원 가능

### 2단계 완료 기준
1. 팀원 판정: 기존 수작업 보고서와 비교해 "다듬으면 바로 제출 가능"
2. 생성 보고서의 모든 수치에 출처·기준일 존재
3. 미리보기와 내보낸 PPT의 시각 일치(기존 parity 검증 통과)

## 3단계 — 속도·사용성

1. **진행률 표시**: 분석 중 카테고리별 체크리스트("지하철 ✓ / 학교 수집 중… / 공원 대기") — 30초~1분 대기의 체감 문제 해소. 재분석은 1단계 캐시로 수초화
2. **주소 검색 무키화**: 무료 도로명주소 API(juso.go.kr)를 기본 검색으로, Naver 키는 선택 보강으로 강등 → 신규 팀원 "가입 즉시 사용"
3. **함정 제거**: ① 주소 미입력 시 분석 버튼 무반응 → 안내 표시 ② 지도 드래그(pan) 시 지하철 오버레이 미갱신 → moveend 갱신 추가 ③ 오버레이 활성 시 무효화되는 지하철 마커 옵션 정리 ④ auth-fetch.ts "/site" 하드코딩 제거
4. **범위 외 명시**: 모바일 반응형(내부 데스크톱 도구), 경로 기반 도보시간, 관리자 대시보드

### 3단계 완료 기준
1. 신규 계정이 API 키 설정 없이 주소 검색→분석→PPT 다운로드까지 완주
2. 분석 중 진행 상태가 카테고리 단위로 표시

## 운영 원칙 (전 단계 공통)

- 코드 작업은 Sonnet 에이전트, 계획·리뷰·런타임 검증은 Fable
- 단계 완료 절차: 구현 → Fable 런타임 검증(실제 구동·스크린샷) → main 머지·푸시 → **배포는 매 회 사용자 승인** → 서버 BUILD_ID 확인 후 완료 기록 (2026-07-03 "배포 착각" 사고 재발 방지)
- 예상 규모: 1단계 1~2세션, 2단계 2~3세션(템플릿 수령 후), 3단계 1~2세션
- 각 단계는 독립 spec→plan→구현 사이클로 진행. 본 문서는 로드맵 상위 설계이며, 단계별 실행 계획은 별도 작성
