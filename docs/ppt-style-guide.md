# Site Analysis PPT Style Guide

## 1. Purpose

This guide defines the visual direction for the app's generated PowerPoint reports. It replaces the current "token variation" mindset with a stricter editorial/report design system.

The generated deck must feel like a premium real-estate site analysis report, not a decorated map screenshot deck.

Primary audience:
- Real-estate developers
- Asset managers
- Investment review committees
- Urban planning and site feasibility teams
- Executives who need quick location judgment

Primary output:
- Editable `.pptx` generated by PptxGenJS
- 16:9 wide report deck
- Map-heavy, data-supported, executive-readable

Secondary output, if added later:
- HTML slide deck with stronger editorial and motion treatment

## 2. Diagnosis Of The Current 10 Styles

The current app already has a strong technical structure:
- `PptDesignConfig` separates frame, title, panel, marker, legend, metric, and composition tokens.
- `PPT_DESIGN_PRESETS` exposes 10 named presets.
- The preview renderer mirrors the PPT generator.
- QA checks ensure preset count, token coverage, and uniqueness.

The weakness is not architecture. The weakness is art direction.

Current styles rely too much on combining decorative tokens:
- frame style
- title style
- panel style
- marker style
- composition style

That creates variety, but not necessarily taste. A style should be a coherent editorial system with a reason to exist, a reference mood, a slide rhythm, type hierarchy, color discipline, and clear rules for map/data treatment.

## 3. Design Principles

### 3.1 Clarity Before Personality

This app is a decision-support tool. Every design choice must make the site, radius, POIs, supply, risk, and recommendation easier to understand.

Avoid:
- decorative lines that do not clarify structure
- heavy HUD elements over satellite maps
- tiny text for visual atmosphere
- too many competing accent colors
- "cool" styles that weaken report credibility

Prefer:
- strong figure-ground separation
- fewer, larger typographic levels
- wide margins around analytical text
- restrained accent color
- map labels with reliable contrast

### 3.2 Editorial, Not Dashboard

Borrow from premium reports and magazines:
- large serif or high-character headlines on cover and section slides
- quiet sans-serif body text
- monospace metadata only for dates, coordinates, source labels, and slide indices
- generous whitespace
- asymmetry with intent
- pull-quote style key findings

Do not borrow from generic AI dashboards:
- purple/indigo gradients
- glowing cards
- gradient text
- neon cyan/magenta palettes
- identical rounded cards everywhere
- emoji as section icons

### 3.3 Map Is The Hero, Data Is The Lens

Most slides start with a real geographic image. The design should frame the map and explain what matters in it.

Rules:
- The base map should stay inspectable.
- Analytical overlays must be legible at projector distance.
- Use panels as lenses, not as opaque blocks covering the map.
- Keep legends small but readable.
- Emphasize the subject site and selected radius more than decorative frame elements.

### 3.4 One Deck, One Visual Voice

A generated deck should not feel like 10 templates stitched together. Each selected preset must control the whole deck consistently:
- cover
- overview map
- score dashboard
- insight summary
- radius/category slides
- development/risk slide
- residential supply slide
- source/disclaimer slide
- closing slide

The preset may vary composition by slide type, but it must preserve the same type, color, spacing, and annotation grammar.

## 4. Required Slide Rhythm

The deck should follow a narrative arc instead of a mechanical export order.

Recommended sequence:
1. Cover: location identity, report date, one-line thesis
2. Executive snapshot: 3-5 strongest findings
3. Full-bleed site map: radius, site marker, primary POI categories
4. Access and mobility: subway, roads, walking catchment
5. Living infrastructure: schools, parks, mountains, daily amenities
6. Development context: maintenance projects, supply, competition, risk
7. Residential supply: nearby projects, dates, scale, price/supply implications when available
8. Recommendation: strengths, watch-outs, decision posture
9. Sources and assumptions
10. Closing / appendix marker if needed

For short decks, keep 6-8 slides. For richer client reports, expand to 10-14 slides by splitting dense category slides rather than shrinking text.

## 5. Typography

### 5.1 Font Roles

PowerPoint compatibility matters, so use practical Korean-safe font stacks.

Recommended default:
- Display / cover title: `Noto Serif KR`, fallback `Batang`, `serif`
- Body / labels: `Noto Sans KR`, fallback `Malgun Gothic`, `Arial`
- Metadata / coordinates / source: `IBM Plex Mono`, fallback `Consolas`, `monospace`

If only one font can be used in generated PPTX, use `Noto Sans KR` and create hierarchy through size, weight, tracking, and whitespace.

### 5.2 Size Rules

Minimum sizes:
- Cover title: 42-56pt
- Section title: 30-40pt
- Slide title: 22-30pt
- Lead insight: 18-24pt
- Body: 13-16pt
- Map labels: 9-12pt
- Legend: 8.5-10pt
- Source/disclaimer: 8.5-10pt

Hard rule:
- Do not use body text below 12pt in the main report.
- Do not let map labels sit directly on satellite imagery without a background, halo, or leader-line callout.

### 5.3 Hierarchy Rules

Every slide should have at most four visible text levels:
1. Title
2. Lead insight or metric
3. Supporting body
4. Metadata/source

If a slide needs more levels, split it.

- beige/cream everywhere without sharp ink contrast

## 7. Map And Overlay System

### 7.1 Satellite Map Treatment

Satellite maps are visually noisy. Every overlay needs its own contrast strategy.

Use one of:
- dark translucent panel: 85-92% visual opacity
- light translucent panel: 88-95% visual opacity
- solid paper panel with subtle transparency only at edges
- label chip with 1-2px high-contrast stroke
- leader-line callout with detached label box

Avoid:
- blur-dependent readability, because PPTX does not reliably support CSS blur
- transparent glass panels below 70% opacity
- thin text directly on map
- low-contrast gray over satellite imagery

### 7.2 POI Marker Rules

Markers should look like report annotations, not app pins.

Recommended marker patterns:
- small filled dot with white ring for normal POIs
- numbered callout disc for ranked POIs
- transit node with line cap for subway stations
- outlined diamond or square for planned/development items
- large site marker with crosshair or target ring

Category colors must remain stable across all styles:
- Subway: amber/gold
- School: blue/teal
- Park/mountain: green/sage
- Residential/apartment: red/rose
- Maintenance/development: cranberry/magenta-red, not neon pink

### 7.3 Label Density

Not every POI deserves a label.

Rules:
- Label the subject site on every map slide.
- Label the top 5-8 most decision-relevant POIs.
- Use unlabeled markers for context.
- For dense subway/POI clusters, use grouped callouts or ranked side panels.
- Never allow labels to create a ring of clutter around the site.

## 8. Layout Grammar

### 8.1 Approved Slide Compositions

Use these compositions as first-class slide types:

1. Full-Bleed Map + Insight Rail
   - Map fills the slide.
   - One side rail contains the finding, 3 supporting facts, and source.

2. Editorial Cover
   - Large title.
   - Small metadata row.
   - Map or aerial image as quiet background crop.
   - One thesis sentence.

3. Score Memo
   - 3-5 large numbers.
   - No more than one chart.
   - Short recommendation line.

4. Evidence Spread
   - Left: map/evidence image.
   - Right: insight, ranked bullets, source.

5. Risk Matrix
   - 2x2 or ranked list.
   - Use color sparingly.
   - Explain the decision implication.

6. Data Ledger
   - Compact table.
   - Clear column alignment.
   - Minimal borders.
   - Source line mandatory.

7. Closing Recommendation
   - One clear decision posture.
   - Three supporting arguments.
   - Watch-out list if needed.

### 8.2 Composition Variety

Do not center every slide. Across the deck, vary:
- left rail
- right rail
- bottom caption bar
- full-bleed image with overlay
- split evidence layout
- data ledger layout
- quiet closing layout

Three slides in a row with the same spatial composition is a design failure.

## 9. Preset System Requirements

Each PPT preset must define more than colors.

Every preset needs:
- name
- intended audience
- visual reference
- palette
- type hierarchy
- map treatment
- marker treatment
- data panel treatment
- best slide types
- forbidden moves
- QA notes

Preset concept descriptions should be specific enough that a designer could recreate the style without seeing the code.

Weak concept:
- "Modern premium report style."

Strong concept:
- "Warm editorial location memo for private investment review: cream paper canvas, serif cover titles, restrained terracotta and sage accents, full-bleed satellite maps framed by quiet paper rails, large single finding per slide."

## 10. Recommended Replacement Presets

The current 10 presets should be replaced or heavily rewritten around these directions.

### 10.1 Executive Location Memo

Audience: executives, investment committee.

Mood: premium consulting memo, quiet but decisive.

Use:
- Research Navy palette
- large sans-serif titles
- narrow metadata rail
- 3-5 strong metrics
- minimal map decorations

Avoid:
- HUD corners
- heavy grid
- more than two accent colors

### 10.2 Editorial Site Brief

Audience: client presentation, proposal, early site storytelling.

Mood: magazine-style real-estate field report.

Use:
- Editorial Ink palette
- serif cover and section titles
- big pull quotes
- image/map spreads
- generous whitespace

Avoid:
- dashboard cards
- tiny bullets
- technical grid backgrounds

### 10.3 Urban Planning Atlas

Audience: planners, architects, feasibility teams.

Mood: civic map atlas and planning sheet.

Use:
- Urban Blueprint palette
- fine grid only on non-map areas
- coordinate-like metadata
- measured linework
- transit and radius diagrams

Avoid:
- luxury gold
- dramatic dark backgrounds
- decorative diagonal lines

### 10.4 Mobility Catchment Report

Audience: TOD, station-area, access analysis.

Mood: transit atlas, wayfinding, catchment logic.

Use:
- restrained transit colors
- route lines with clear hierarchy
- station labels with strong halos
- side panels for travel implications

Avoid:
- rainbow subway decoration
- over-labeling every station
- cartoonish transit iconography

### 10.5 Living Environment Brief

Audience: residential planning, sales strategy, livability review.

Mood: calm park/lifestyle report.

Use:
- Civic Neutral palette
- sage and warm brown accents
- park/mountain accessibility panels
- soft but readable markers

Avoid:
- low-contrast green-on-green
- organic decoration that hides data
- lifestyle fluff without evidence

### 10.6 Supply And Competition Ledger

Audience: sales, development, acquisition teams.

Mood: investment ledger, compact but readable.

Use:
- Research Navy or Asset Graphite palette
- table-first supply slides
- ranked residential project callouts
- date and scale emphasis

Avoid:
- dense table squeezed onto one slide
- unreadable 8pt rows
- decorative cards around every row

### 10.7 Development Risk Memo

Audience: development, construction, risk review.

Mood: controlled warning system, not alarmist.

Use:
- cranberry/risk accent
- 2x2 risk matrix
- staged project timelines
- clear watch-out labels

Avoid:
- neon red/magenta
- "war room" gimmicks
- excessive terminal styling

### 10.8 Premium Asset Brochure

Audience: high-value site pitch, investor-facing brochure.

Mood: high-end asset brief.

Use:
- Asset Graphite palette
- gold as a thin accent only
- full-bleed hero map/image
- quiet keylines
- large location identity

Avoid:
- gold overload
- dark text blocks with poor contrast
- luxury style on data-dense slides

### 10.9 Public Evidence Report

Audience: public agency, internal compliance, source-heavy review.

Mood: neutral evidence document.

Use:
- Civic Neutral palette
- high source visibility
- conservative tables
- clear assumptions slide

Avoid:
- cinematic covers
- subjective adjectives
- unclear data provenance

### 10.10 Minimal Decision Note

Audience: quick internal decision.

Mood: monochrome memo with one accent.

Use:
- black/white/one accent
- maximum content discipline
- one insight per slide
- few shapes

Avoid:
- decorative frames
- multiple chart styles
- more than three visible text levels

## 11. Polish And QA Checklist

Every generated deck should pass this before download or before a "final" state is claimed.

### 11.1 Structural Completeness

- Title slide exists.
- Recommendation or closing slide exists.
- Source/assumption slide exists.
- Deck has a clear story arc, not only category exports.
- Slides over 8 content pages include section rhythm.

### 11.2 Metadata

Where supported:
- document title is set
- author/app name is set
- subject/keywords are set
- generated date is visible

### 11.3 Source Lines

Every data-heavy slide needs a source line:
- map tile/source
- POI source
- date of generation
- radius and search assumptions where relevant

Use:
- 8.5-10pt
- muted color
- bottom-left or bottom-right
- consistent location across deck

### 11.4 Typography Consistency

- No more than 4 main font sizes per deck, excluding map labels.
- Body text remains 13-16pt.
- Source lines remain 8.5-10pt.
- Headings are consistent by slide type.
- Korean text does not overflow boxes.

### 11.5 Color Consistency

- One palette per deck.
- Category colors remain stable.
- Accent color is not used for every object.
- Risk colors are reserved for risk.
- No off-palette colors are introduced by one-off fixes.

### 11.6 Layout Consistency

- Common slide types start content at consistent positions.
- Margins are respected.
- Legends do not jump randomly slide to slide.
- Similar data panels use the same width and typography.
- Tables align numerics with tabular style.

### 11.7 Map Legibility

- Subject site is obvious within 2 seconds.
- Radius rings are visible but not dominant.
- Top POIs are readable.
- Dense POIs do not create label clutter.
- Satellite background does not overpower text.

## 12. Implementation Notes For This Codebase

The current `PptDesignConfig` shape is useful and should be kept, but preset authoring should become stricter.

Recommended additions:
- Add a `voice` or `audience` field to `PptDesignPreset`.
- Add a `bestFor` field to guide automatic preset selection.
- Add a `forbidden` or `guardrails` field for QA.
- Add `sourceStyle`, `tableStyle`, and `sectionDividerStyle` tokens if the deck expands.
- Add preset-level text scale tokens for cover, section, body, labels, and sources.

Recommended QA improvements:
- Verify source line presence on data slides.
- Verify no body text below 12pt.
- Verify no forbidden colors in presets.
- Verify each preset has a non-generic concept description.
- Verify generated slides include source/assumption slide.
- Add screenshot or canvas-based visual checks for text clipping.

## 13. Rule Of Thumb

If a style can be described only by its colors and decorative frame, it is not a style.

A real preset should answer:
- Who is this for?
- What decision does it support?
- What visual tradition does it borrow from?
- How does it treat maps?
- How does it treat evidence?
- What does it refuse to do?

Only presets that can answer those questions should remain in the app.
